**✨ 정리**

| 방식              | 예시              | 내가 관리하는 정도   | 커스터마이징  | 성능/비용                |
| ----------------- | ----------------- | -------------------- | ------------- | ------------------------ |
| Self-hosting      | Wowza, NGINX RTMP | 내가 다 함           | 거의 무제한   | 높은 자유도, 높은 복잡도 |
| 클라우드 스트리밍 | AWS IVS, Mux      | 플랫폼이 관리        | API로 가능    | 편리, 유료               |
| CDN + VOD         | S3 + CloudFront   | 중간쯤 (파일 관리만) | 가능          | 매우 저렴, 안정적        |
| iframe 임베딩     | YouTube, Vimeo    | 거의 안 함           | 불가능/제한적 | 공짜지만 통제 불가       |

> VOD
>
> - **Video On Demand** (주문형 영상)
> - 넷플릭스처럼 → 사용자가 요청할 때 영상을 “다운받는 것처럼 보여주지만 실은 스트리밍”
> - 즉, **실시간 라이브가 아닌 영상 재생** 전부는 거의 다 VOD라고 보면 됨
>   영상(mp4, m3u8 등)을 S3, Firebase, GCP에 업로드
>   CloudFront, [Bunny.net](http://bunny.net/) 같은 CDN을 연결
>   🎯 **즉, CDN + VOD 방식은 가장 단순하고 저렴한 스트리밍 형태입니다.**

```
  <video controls>
    <source src="<https://cdn.mysite.com/intro.mp4>" type="video/mp4" />
  </video>
```

> CDN + VOD이란?
> 📦 CDN + VOD = “파일을 미리 올려두고 스트리밍처럼 제공”
>
> - 영상 같은 큰 파일을 전 세계 서버에 **미리 분산**시켜서 사용자에게 빠르게 제공
> - `https://cdn.example.com/videos/intro.mp4` 이런 주소로 빠르게 재생됨

# CDN,VOD는 streaming이 아닌건가?

** CDN + VOD 방식은 어떤 "스트리밍"?**

- CDN에서 mp4 또는 HLS 파일을 **조각내서 조금씩 다운로드하며 재생**
- 사용자가 요청한 시점의 영상만 받아오고, 나머지는 필요할 때 받아옴
- 즉, 넷플릭스나 유튜브처럼 작동함
  📌 **이건 On-Demand Streaming (VOD 스트리밍)** 이라고 부릅니다.

| 구분          | CDN + VOD                | Wowza, IVS                     |
| ------------- | ------------------------ | ------------------------------ |
| 스트리밍 타입 | 🎬 On-Demand (요청 기반) | 🔴 Live (실시간 방송)          |
| 데이터 원본   | 미리 업로드된 파일       | 현재 송출 중인 비디오          |
| 지연 시간     | 없음 또는 3~10초         | 1~5초 (HLS), 1초 이하 (WebRTC) |
| 플레이어      | `<video>` / `hls.js` 등  | `hls.js`, WebRTC 등            |
| 업로드 방식   | 수동 (S3, Firebase 등)   | 인코더 → RTMP Push             |
| 비용 구조     | 저장 용량 + CDN 트래픽   | 송출 시간, 시청자 수           |

<br />
**wowza engine, wowza cloud, AWS IVS 비교**

| 기능           | Wowza Streaming Engine | Wowza Cloud            | AWS IVS     |
| -------------- | ---------------------- | ---------------------- | ----------- |
| 방식           | 직접 설치, 내 서버     | Wowza 운영             | AWS 운영    |
| Live Streaming | ✅                     | ✅                     | ✅          |
| VOD 지원       | ✅ (mp4 → HLS)         | ❌ 직접 저장 구현 필요 | ❌          |
| 유연성         | 매우 높음              | 보통                   | 낮음        |
| 관리 편의      | 낮음 (내가 관리)       | 높음                   | 매우 높음   |
| 비용           | 설치형/고정            | 구독 or 사용량         | 사용량 기준 |

Wowza Streaming Engine에서는 **VOD(Video On Demand)** 파일을 **특별한 인코딩 없이**,

그냥 mp4 파일만 서버에 올려두면 자동으로 **HLS(HTTP Live Streaming) m3u8** 포맷으로 변경해준다.

```
<http://your-wowza-server:1935/vod/mp4:myvideo.mp4/playlist.m3u8>
```

서버에서는 저렇게 wowza로 스트리밍으로 주고 그걸가지고 api로 변환된 파일 .m3u8를 클라이언트로 던져줌 그럼 이걸 클라이언트에서 video.js같은 라이브러리로 비디오 재생

예시 vod/_definst_/mp4:0000564/A0005644690825_720p.mp4/playlist.m3u8
// https://wowza.tvcf.co.kr:1443/

# 전체 동작 흐름 요약

**- 서버 측 (Wowza + API)**

1. **MP4 원본 파일**이 서버(Wowza)에 저장됨

   예: `/vod/_definst_/mp4:0000565/A0005655952BDD_720p.mp4`

2. **Wowza가 HLS 형식으로 변환**
   - `.m3u8` → HLS 마스터/미디어 플레이리스트
   - `.ts` → 분할된 영상 조각 (세그먼트)
3. **API 서버가 클라이언트에 `.m3u8` URL을 전달**

   예:

   ```json
   json
   {
     "videoUrl": "https://wowza.tvcf.co.kr:1443/vod/_definst_/mp4:0000565/A0005655952BDD_720p.mp4/index.m3u8"
   }

   ```

**- 클라이언트 측 (video.js / hls.js)**

1. **프론트는 API로부터 `.m3u8` URL을 받아 video.js에 전달**

```tsx
tsx;
videojs(videoRef.current, {
  sources: [
    {
      src: videoUrlFromApi,
      type: 'application/x-mpegURL',
    },
  ],
});
```

1. **video.js는 `.m3u8`을 fetch → `.ts` 세그먼트를 순차적으로 다운로드**
2. **Media Source Extensions (MSE)를 통해 `<video>` 태그로 버퍼링 및 재생**

```jsx
[ MP4 저장 (Wowza) ]
        ↓
[ HLS 변환 (m3u8 + ts) ]
        ↓
[ API: m3u8 URL 전달 ]
        ↓
[ video.js (브라우저) ]
        ↓
[ 재생: <video> + MSE + 버퍼 ]
```

**요점 먼저:**

> "스트리밍으로 전환된 파일"이란 건
>
> 🎞️ **Wowza가 MP4를 작은 단위의 `.ts` 조각(세그먼트)로 분할하여 실시간 또는 사전 처리 방식으로 제공하는 것**입니다

# 스트리밍 전환 방식 2가지

| 방식              | 설명                                                                                                 | 대표 상황                   |
| ----------------- | ---------------------------------------------------------------------------------------------------- | --------------------------- |
| **VOD 스트리밍**  | ✅ MP4를 미리 서버에 저장하고요청이 들어오면 **on-demand로 `.ts` 세그먼트를 실시간으로 쪼개서 제공** | TVCF처럼 사용자 업로드 영상 |
| **Live 스트리밍** | ✅ 카메라 → 인코더 → Wowza로 **실시간 ingest** 후**매 2~6초 단위로 `.ts` 조각 생성**                 | 생중계                      |

**VOD의 경우**

- **MP4가 저장된 상태에서, 요청이 들어오면 Wowza가 필요할 때마다 `.ts` 조각을 실시간으로 생성함**
- 즉, **미리 다 만들어놓는 게 아니라 → 요청이 있을 때 동적으로 만든다 (transmuxing)**
- 그래서 `.ts` 파일들을 보면 뒤에 `media_xxxxx_0.ts`, `media_xxxxx_1.ts` 이런 식으로 이름이 붙는 것

> 📌 "스트리밍 파일"이란 게 파일 자체로 생성되는 게 아니라, 플레이어가 요청할 때마다 메모리 상에서 잘라서 보내는 방식임
> ![](/images/video-hls/image.png)

**▶ 각각의 역할은 다음과 같습니다:**

| 파일명                 | 설명                                                                         |
| ---------------------- | ---------------------------------------------------------------------------- |
| **`playlist.m3u8`**    | 👉 **마스터 플레이리스트**→ 여러 화질(예: 1080p, 720p 등)의 chunklist를 참조 |
| **`chunklist_*.m3u8`** | 👉 **미디어 플레이리스트 (720p 한 화질 전용)**→ 실제 `.ts` 세그먼트들을 나열 |
| **`media_*.ts`**       | 👉 **HLS 세그먼트 조각 (실제 영상)**                                         |

<br />
**✅ 마스터/미디어 .m3u8은 언제 나뉘냐?**

| 파일               | 역할                      | 언제 생성?                                                       |
| ------------------ | ------------------------- | ---------------------------------------------------------------- |
| `playlist.m3u8`    | 마스터 리스트             | 여러 화질이 있을 때 자동 생성                                    |
| `chunklist_*.m3u8` | 개별 화질용 미디어 리스트 | **요청된 화질별로 on-demand 생성**                               |
| `.ts`              | 미디어 세그먼트           | `chunklist`에서 순서대로 요청될 때마다 실시간 생성 (혹은 캐싱됨) |
